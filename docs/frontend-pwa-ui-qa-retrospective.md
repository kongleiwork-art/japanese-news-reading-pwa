# 前端问题沉淀与技术复盘

## 背景

本次项目是一个面向中文用户、通过日语新闻学习日语的移动端轻量 PWA。开发过程中，工作重心从“高保真还原 UI”逐步转向“交互稳定性、状态一致性、上线前验收”。

这份文档沉淀本次前端迭代中暴露出来的主要问题、根因、修复方式，以及后续项目应直接复用的技术规则。

## 一、问题总览

### 1. 交互层不稳定

- 新闻详情页点击单词，词条弹层最初无法稳定打开
- 单词本页点击单词与新闻详情页行为不一致
- 复习页卡片翻面依赖本地状态，在当前浏览器环境下表现不稳定

### 2. 布局容器不统一

- 详情页曾出现“桌面宽屏化”，没有保持移动端画布比例
- 固定定位的弹层、底部栏与页面容器基准不一致
- 某些元素被父级 `overflow` 或固定布局裁切

### 3. 数据层与页面层脱节

- 首页卡片并非全部都对应真实详情页
- 分类值、频道值、文案标签曾出现映射不一致
- 可点击词的表层 UI 与底层词条数据不是统一来源

### 4. 原型式按钮过多

- 一些头部图标、设置项、模式按钮看起来能点，但实际上没有跳转或状态意义
- 影响整体验收感，像“做出来了”，但不像“可上线产品”

### 5. 文案与编码污染

- 代码里残留了大量乱码/脏文本
- 中文 UI 与日文内容没有完全分层
- 后续修改容易出现“一页正常、另一页仍旧异常”的问题

### 6. 验收方式过于页面化

- 最开始更像在逐页修样式，而不是按“路由矩阵 + 状态矩阵”做验收
- 导致同一类问题在不同页面反复出现

## 二、核心根因

### 1. 过度依赖纯前端瞬时状态

最典型的是词条弹窗和复习翻面。它们最初依赖组件内部状态或点击事件链路，在当前 in-app browser 环境下不够可靠，出现了：

- 点击命中正常，但状态没更新
- 状态更新了，但弹层被裁切或没显示在可视区
- 同样的交互逻辑在不同页面表现不一致

### 2. 先做“视觉像”，后补“状态真”

很多页面在视觉上已经接近设计稿，但交互模型并没有同步升级成可验收的产品结构，例如：

- 首页新闻卡有入口但无详情目标
- 按钮有视觉但无真实行为
- 页面可以展示，但返回/关闭/选中态不完整

### 3. 缺少统一的数据真源

频道、分类、文章、词条、复习状态曾分别散落在页面逻辑里，导致：

- 枚举值与显示文案不一致
- 一处修正后，另一处继续引用旧数据
- 页面间相同概念表现不统一

## 三、最终采用的修复策略

### 1. 用 URL 状态替代脆弱的本地瞬时状态

这是本次最关键的技术决策。

落地方式：

- 词条详情改为 `?word=...`
- 复习翻面改为 `?side=back`
- 首页频道和分类继续使用 query 参数

收益：

- 状态可复现、可回退、可分享
- 在不稳定的浏览器环境里仍能稳定工作
- 页面刷新后状态仍可恢复

### 2. 强制所有主页面遵循移动端容器基准

统一规则：

- 所有主要页面固定在移动端画布宽度内
- 弹窗、底部导航、固定操作栏都以同一容器逻辑对齐
- 避免“详情页独自撑满桌面”的问题

### 3. 把“看起来能点”的控件收敛成“真的能点”

修复原则：

- 真按钮必须有行为
- 没有真实行为的控件，不允许继续维持强交互视觉
- 未实现的产品动作优先改成内部锚点或真实页面跳转

### 4. 重写共享数据层

统一了：

- 分类枚举
- 频道定义
- 文章列表
- 文章详情
- 单词本分组
- 复习数据
- 我的页统计数据

收益：

- 首页、详情页、单词本、复习页之间的链路一致
- 所有可点入口有真实数据承接
- 后续接入 NHK 真数据时有明确适配面

### 5. 从“逐页修”升级为“路由矩阵验收”

验收对象不再只是页面截图，而是：

- 首页默认态
- 首页原文频道态
- 首页分类过滤态
- 文章详情默认态
- 文章详情词条态
- 单词本默认态
- 单词本词条态
- 复习正面态
- 复习背面态
- 复习背面 + 词条态
- 我的页

## 四、这次沉淀出的技术规则

### 规则 1

对于移动端 PWA 原型，只要交互是核心学习链路的一部分，就优先考虑 URL 驱动状态，而不是只靠组件内 `useState`。

### 规则 2

首页任何一张内容卡只要能点，就必须有真实详情页或真实跳转目标。

### 规则 3

弹层、抽屉、固定底栏这类浮层组件，必须先验证容器宽度、定位基准和裁切关系，再谈动画和视觉。

### 规则 4

所有分类、频道、词条、文章源必须来自共享数据层，不能在页面里散写临时字符串。

### 规则 5

一个按钮要么是真行为，要么降级成弱提示，不允许保留“像能点但其实没逻辑”的中间态。

### 规则 6

正式验收必须覆盖“页面 + 状态”组合，而不是只看单个页面默认态。

## 五、建议的后续技术路线

### 第一阶段

继续保持当前 URL 驱动状态模型，补齐以下真实能力：

- 收藏文章
- 收藏单词
- 复习结果持久化
- 设置项本地存储

### 第二阶段

接入真实内容源：

- NHK NEWS WEB EASY
- NHK NEWS WEB

并为双源建立统一的标准化文章模型。

### 第三阶段

在稳定版本上再做增强交互，而不是反过来：

- 弹窗进入/退出动画
- 更高级的卡片翻转动效
- 搜索与筛选联动
- 离线缓存与 PWA 安装体验

## 六、结论

这次前端迭代最有价值的收获，不是单个页面做得更像设计稿，而是把项目从“高保真原型”推进成了“有稳定状态模型的可验收前端”。

真正需要复用的不是某个样式细节，而是这套方法：

- 先稳定状态链路
- 再统一数据层
- 再做视觉验收
- 最后才做高级动效

后续只要是移动端内容型 PWA、学习类产品、或需要在受限浏览器环境中工作的前端原型，都应优先复用这套规则。
